      Domain layer placement rules
This is the layer you protect hardest. It must remain framework-agnostic so it can be tested in isolation and survive infrastructure changes.
domain/
  aggregates/
    <aggregate-name>/
      <aggregate-root>.ts
      entities/
      value-objects/
      events/
      invariants/
  entities/
    <entity-name>.ts
  value-objects/
    <value-object-name>.ts
  services/
    <domain-service-name>.ts
  events/
    <domain-event-name>.ts
  specifications/
    <specification-name>.ts
  factories/
    <factory-name>.ts
  repositories/
    <repository-port>.ts
  exceptions/
    <domain-error>.ts
  policies/
    <domain-policy>.ts
aggregates/<aggregate-name>/ gets first-class folders because aggregates are the primary consistency boundary. Nesting aggregate-specific entities and value objects inside the aggregate folder makes invariants discoverable and prevents accidental re-use of internals across aggregates.
domain/entities/ is only for entities shared by more than one aggregate inside the same bounded context. If an entity only makes sense inside one aggregate, keep it inside that aggregate folder.
domain/value-objects/ is only for context-wide value objects. Aggregate-private value objects stay in the aggregate. This avoids fake reuse that later weakens boundaries.
domain/events/ stores pure domain events emitted by aggregates or domain services. These are internal business facts. They are not Kafka payloads yet.
domain/repositories/ contains repository interfaces only. Infrastructure implements them.
6. Application layer placement rules
The application layer is where use cases live. Every state-changing use case gets a command and a handler. Every read use case gets a query and a handler. Longer flows get workflows or sagas.
application/
  commands/
    create-<thing>.command.ts
    update-<thing>.command.ts
  command-handlers/
    create-<thing>.handler.ts
    update-<thing>.handler.ts
  queries/
    get-<view>.query.ts
    list-<view>.query.ts
  query-handlers/
    get-<view>.handler.ts
    list-<view>.handler.ts
  use-cases/
    <verb>-<noun>/
      request.ts
      response.ts
      handler.ts
  services/
    <context>-application-service.ts
  workflows/
    process-managers/
    sagas/
  policies/
  ports/
    repositories/
    gateways/
    event-bus/
    storage/
    identity/
  dto/
  mappers/
  validators/
  authorization/
  tenancy/
If the team prefers explicit use-case folders over command-query split, keep both by letting use-cases/<verb-noun>/ re-export the command, DTO, and handler for discoverability. The important point is one predictable home for every use case.
application/services/ should stay thin. These are orchestration facades for a group of related use cases, not giant transaction scripts.
application/ports/ defines what the application needs from the outside: repositories, event bus, email, object storage, clock, identity context, or upstream context gateways.

    tracing/

persistence/postgres/schemas/ holds table or schema declarations. orm/ holds query-builder or ORM setup if you use one. repositories/ contains concrete repository implementations. mappers/ translates persistence rows to domain objects. queries/ is for optimized read SQL that should not leak into aggregate repositories.
messaging/kafka/outbox/ is critical. Domain events should first be stored transactionally, then converted into integration events for Kafka publication. This protects you from lost events.
integrations/<external-system>/ keeps each upstream or downstream dependency isolated. Every integration gets its own client, mappers, auth strategy, and retry policy so failures stay local.

shared/kernel/ is the smallest allowed shared surface: IDs, Result or Either types, base errors, guard helpers, and simple primitives. If you put business language here, you will create a hidden distributed monolith.

shared/ui/ is the Angular design system and cross-app UI primitives. Keep domain-specific feature components out of it.

13. Kafka and event flow structure
Use two event concepts: domain events inside the bounded context and integration events published to Kafka. Never publish raw domain objects directly.
Flow:
  domain aggregate emits DomainEvent
    -> application handler persists state
    -> outbox record created in same transaction
    -> worker reads outbox
    -> integration event mapper builds event contract
    -> kafka producer publishes
    -> consumer in another app or context handles it
    -> projection or workflow updated
Location
Contents
Why
domain/events/
Pure business facts
Keeps domain language clean and testable.
contracts/events/
Versioned integration payloads
Stable cross-context and cross-service contracts.
infrastructure/messaging/kafka/
Producers, consumers, serializers, topic bindings
Concrete transport mechanics isolated from business code.
data/kafka/schemas/
Avro or JSON schema definitions
Governed compatibility and reviewable event evolution.

14. PostgreSQL structure and ownership
Even with one shared Postgres cluster and a shared database, act as if each bounded context owns its own schema slice. The easiest practical version is one database with schema namespaces per context, or at minimum naming conventions and migration folders per context.
Recommended logical ownership:
  whizard_identity_access.*
  whizard_governance.*
  whizard_capability_framework.*
  whizard_certification_progression.*
  whizard_shared.*   (only truly cross-cutting technical tables such as outbox, migration metadata, idempotency)
Repository implementations live inside each bounded context. Cross-context reads for dashboards should use read models or query services, not direct writes into another context schema.

16. Naming conventions
Use kebab-case for folders, file names, and Nx project names.
Use singular names for aggregates and entities (certification-record, not certification-records).
Use explicit suffixes for adapters: .repository.ts, .producer.ts, .consumer.ts, .gateway.ts, .mapper.ts, .handler.ts.
Use version suffixes for published contracts: customer-created.v1.ts.
Keep public entry points in src/index.ts and src/public-api.ts only.
17. Guardrails for vibe coding
Nx project tags and enforced dependency constraints so presentation depends on application, not private internals of another context.
ESLint import boundaries to block cross-context private imports.
Template generators for new bounded contexts, aggregates, commands, and handlers.
PR checklist requiring correct location, ports before adapters, versioned contracts, tests, and docs updates.
Architecture tests that fail if infrastructure imports appear inside domain folders.
18. First implementation sequence
Create the top-level folders and Nx projects first, before writing business logic.
Establish shared/kernel, shared/application, and shared/infrastructure packages.
Generate one bounded context template and copy the structure consistently.
Implement transaction, outbox, idempotency, auth context, and tenancy context before feature work.
Add one vertical slice end to end (for example: create certification level) to validate the structure.
Only then scale to the remaining bounded contexts.

Here we have made an attempt to optimize the structure for predictability, enforceability, and change safety. The right compromise for this platform is a modular monolith with strict bounded-context folders, explicit application use cases, pure domain models, Postgres ownership by context, and Kafka only through outbox-backed integration events. That gives us enough structure to move fast without turning the repo into a maze.
